1.3 분석 대상 선정 이유

결론: 이 프로젝트의 목적은 LLM을 Fine-tuning하는 것이며, 이를 위해서는 'Instruction(지시)'과 'Output(결과)' 쌍으로 이루어진 "라벨링 데이터"가 필수적이기 때문입니다.

'원본 대화 (Source)'는 단순히 고객과 상담사의 대화 로그(`content`)입니다. 이 자체만으로는 모델을 "어떻게" 학습시킬지 알 수 없습니다.

'라벨링 데이터 (Label)'는 이 원본 대화를 기반으로 `"Instruction(지시)"`과 `"Output(정답)"`을 가공한 **Fine-tuning 전용 데이터셋**입니다. (예: `Instruction`: "이 대화에서 고객이 원하는 것을 찾아", `Output`: "성인 2명, 아이 2명")

따라서 이번 EDA의 목적은 LLM을 훈련시키기 위한 **'Instruction Tuning'용 데이터의 특징'**을 파악하는 것이므로, 205,711건의 '라벨링 데이터'를 분석의 중심으로 삼았습니다.

1.4 데이터 로딩 이슈

Colab-Drive 환경에서 한글 파일 이름이 **유니코드 NFD(자모 분리)** 방식으로 저장되어, Python이 파일 필터링에 실패하는 문제를 `unicodedata.normalize('NFC', ...)`로 해결했습니다.

🎯 Task별 심층 분석

4.1 질의응답 (Q&A) - ⭐️ 프로젝트의 핵심

결론: 이 태스크는 LLM에게 "주어진 맥락 내에서 사실을 찾아 짧게 생성"하도록 훈련시키는 'Instruction Tuning'용 데이터입니다.

모델 유형 (가장 중요)

발견: 답변의 **70.3%**가 원본에 없는 단어로 재구성된 **생성형(Abstractive)**이었습니다. (예: 원본 "만 오천 원입니다." → 답변 "15,000원"). 이는 `KoBERT-QA` 같은 추출형 모델로는 풀 수 없음을 의미합니다.

Instruction (질문) 분석

발견: 사용자의 "자연어 질문"이 아닙니다. '언제', '얼마나'가 아닌, **'고객은', '고객이', '민원인은'** 등이 대부분입니다.

해석: 이는 모델에게 "이 대화에서 '고객이' 한 말을 찾아"라고 지시하는 **Instruction(지시) 템플릿**입니다.

Output (답변) 분석

발견: "짧은 사실(Fact)"의 형태를 띱니다.

근거: 평균 16자, 99%의 답변이 **62자** 이내입니다. (예: "성인 2명, 아이 2명")

4.2 분류 (Classification)

해석: '민원인', '단일 요건 민원', '만족' 등 화자, 민원 유형, 감정, 주제 등 **다양한 종류의 레이블**이 혼재되어 있습니다.

4.3 요약 (Summarization)

해석: Q&A(평균 16자)보다 훨씬 긴 **평균 154자**의 길이를 가집니다. 전형적인 **추상적 요약(Abstractive Summarization)** 태스크입니다.

🚀 결론: 구현 전략

전략 1: 모델 선정 (Model Selection)

방향: **Seq2Seq (Encoder-Decoder) 생성 모델**

근거: Q&A의 70.3%가 생성형(Abstractive)이었고, 요약 태스크 역시 생성형입니다. `KoBERT`나 `KoELECTRA` 같은 Encoder-only 모델은 이 문제를 풀 수 없습니다.

권장 모델: **T5** (예: `KETI/ke-t5-base`), **BART** (예: `digit82/kobart-base`), 또는 **Llama/Gemma/EEVE** 등 생성형 LLM.

전략 2: 훈련 방식 (Training Strategy)

방향: **멀티태스크 학습 (Multi-Task Learning)을 위한 프롬프트 엔지니어링**

근거: 동일한 원본 대화(content)에 대해 3가지 다른 레이블(Q&A, 분류, 요약)이 존재하므로 Multi-Task Learning에 완벽한 조건입니다.

구현: 모델이 3가지 임무를 구분하도록 **Prefix(접두사) 또는 Prompt(지시어)**를 사용하여 입력 형식을 통일합니다.

Task Prefix (Prompt) 예시 Input 예시 Output
Q&A `qa:` `qa: 고객은 언제 방문했나요? content:...[대화 본문]...` `▲월 ▲▲일`
분류 `classify:` `classify: content:...[대화 본문]...` `단일 요건 민원`
요약 `summarize:` `summarize: content:...[대화 본문]...` `고객은 가족 여행으로...`

전략 3: 데이터 전처리 (Preprocessing)

  • Noise 제거: EDA 3단계에서 발견한 '담사', '손님' 및 '고객:', '상담사:' 같은 화자 태그를 정규표현식으로 모두 제거하여 순수 텍스트만 남깁니다.
  • Truncation (잘라내기): EDA 2단계에서 확인한 99% 지점(약 2,548자)을 기준으로, 선정한 모델의 최대 토큰 길이를 `max_length`로 설정하고, 이보다 긴 1%의 데이터는 앞부분을 잘라냅니다.
